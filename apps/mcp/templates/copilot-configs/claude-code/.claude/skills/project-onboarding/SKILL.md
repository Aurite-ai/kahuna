# Project Onboarding

## Purpose

Capture essential business context for new projects. This quick 2-3 question flow establishes what success looks like for this specific project.

## When to Use

Trigger this skill when the user:
- Says "set up project context" or "describe my project"
- Says "onboard my project" or "help me onboard"
- Says "update project context" (re-run for existing projects)
- Responds to the kahuna_initialize hint about project context

## Process

### Step 0: Check for Organization Context

Before asking project questions, check if organization context exists:

1. Use `kahuna_ask` to query: "What is our organization context? What industry and priorities?"
2. If org context is found → Proceed to Step 1 (use it to tailor questions)
3. If org context NOT found → Tell the user:

> I don't have any organization context yet. Let me ask a few quick questions about your organization first, then we'll set up this project.

Then **trigger the org-onboarding skill** (ask the first org question) and return here after it completes.

### Step 1: Ask About the Problem

Ask the user (optionally tailored if org context exists):

> **What problem does this project solve? Who experiences this problem?**

If org context mentions a specific industry (e.g., healthcare), you can tailor:
> **What [healthcare/patient/provider] problem does this project solve?**

Wait for their response.

### Step 2: Ask About Users

Ask the user:

> **Who will use this? What do they need to accomplish?**

If org context mentions solo developer, you can tailor:
> **Is this for your own use, for clients, or for end users?**

Wait for their response.

### Step 3: Ask About Success

Ask the user:

> **What does success look like? How will you know this project worked?**

If org context mentions compliance as a priority, you can add:
> **How does this support your compliance goals?**

Wait for their response.

### Step 4: Confirm Summary

Before storing, show the user a summary of what was captured:

> **Here's what I captured for this project:**
> - **Problem:** [brief summary of their answer]
> - **Users:** [brief summary of their answer]
> - **Success criteria:** [brief summary of their answer]
>
> Does this look right? (say "yes" to continue, or correct anything)

If they say "yes", "looks good", "correct", etc. → proceed to Step 5.
If they provide corrections → update the relevant field and confirm again.

### Step 5: Synthesize and Store

**Step 5a: Generate the filename:**
1. Get the current working directory path (e.g., `/Users/dev/projects/my-app`)
2. Generate a short hash (6 characters) from the path - use the first 6 hex characters of an MD5 or similar hash
3. Filename: `/tmp/kahuna/project-context-{hash}.md` (e.g., `/tmp/kahuna/project-context-a1b2c3.md`)

This ensures each project gets its own context file without collisions.

**Step 5b: Create the file** at the path above (create the `/tmp/kahuna/` directory if it doesn't exist). The document should:

1. **Synthesize, don't transcribe** - Capture the essence in clear, structured prose
2. **Preserve their language** - Use their terminology and phrasing where it adds clarity
3. **Keep it concise** - This is reference material, not documentation
4. **Reference org context** - Note alignment with org priorities where relevant

Use this template:

```markdown
# Project Context

**Project:** [infer project name from conversation or working directory]
**Created:** [current date]

## Problem Statement

[Synthesize their answer to Question 1 into 2-3 sentences]

## Target Users

[Synthesize their answer to Question 2 into 2-3 sentences]

## Success Criteria

[Synthesize their answer to Question 3 into 2-3 sentences, as bullet points if they mentioned multiple criteria]

---

*Generated by Kahuna project onboarding | Update with "update project context"*
```

### Step 6: Store in Knowledge Base

**Step 6a:** Call `kahuna_learn` to store the document in the knowledge base:

```
kahuna_learn(
  paths=["/tmp/kahuna/project-context-{hash}.md"],
  description="Project business context and success criteria"
)
```

Replace `{hash}` with the actual hash generated in Step 5a.

**Step 6b:** After `kahuna_learn` succeeds, delete the temporary file `/tmp/kahuna/project-context-{hash}.md`.

### Step 7: Confirm

Tell the user:

> I've captured your project context. This will help me make recommendations aligned with your goals.
>
> You can update this anytime by saying **"update project context"**.

## Notes

- **Keep it quick** - Users should be done in under 2 minutes
- **Don't over-probe** - Accept brief answers; context accumulates over time
- **Natural conversation** - This isn't a form, it's a dialogue
- **Idempotent updates** - Running again from the same directory replaces that project's context
- **Org-aware** - Questions adapt based on organization context when available
- **Path-based uniqueness** - Each project directory gets its own context file via path hash
